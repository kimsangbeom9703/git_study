2. 버전 만들기
	스테이지와 커밋 이해하기 (깃 버전 만드는 단계)

	작업트리 , 스테이지 , 저장소 (리포지토리)

		작업트리 : 작업디렉터리 라고도 하고 현재 눈에 보이는 폴더를 작업트리라고 한다.

		스테이지 : 버전으로 만들 파일이 대기 하는 곳 ex) 작업트리에서 2개의 파일을 수정 했는데 1개만 버전으로 만드려면 1개만 스테이지에 넘겨주면 된다

		저장소   : 버전으로 만들어서 저장하는 곳.
		

	1. git status
		현재 상태를 확인
	2. vi text.txt
		테스트 파일 생성
	3. git add text.txt
		스테이지 로 이동
	4. git commit -m '메세지'
		커밋 
	5. git status 
		확인
	6. git log
		로그 확인

	기본적인 순서고 중간 중간 git status 를 입력하여 상태를 확인한다.

	스테이징과 커밋 한꺼번에 처리하기
		git commit -am "메세지"


3. 커밋 내용 확인하기
	git diff (변경사항 확인하기)
		커밋한 파일을 수정후 git status 를 입력하면 modified 로 어떤 파일이 수정 되었는지 나온다.

		최신 저장소에 있는 내용과 현재 파일이 다른지 확인하는 명령어

		git diff

4. 버전 만드는 단계마다 파일 상태 확인하기
	tracked 파일과 untracked 파일
		말대로 추적하고 있는 파일과 아닌 파일인데 깃은 한번이라도 커밋한 파일은 수정여부를 추적하고 있어 tracked 파일 이라고 하고
		버전관리를 하지 않은 파일은 untracked 파일 이라고 한다.
	
	커밋에 관련된 파일 까지 함께 보려면 git log --stat 를 입력한다.

	방금 커밋한 메세지 수정하기
	git commit --amend


* 중간 정리 
	파일 상태 변화 
		untracked  -> unmodified -> modified -> staged 
		-------------------스테이징-------------------->
		                   --------파일수정---->
										-----스테이징-->
						<----------------커밋-----------
	한번에 커밋하기
		git commit -am "메세지" 사용하면 될듯하다.

6. 작업 되돌리기
	1.	가장 최신 버전으로 되돌리기 (스테이징 전)
		git checkout -- 파일
	
	2.	스테이징 되돌리기
		git reset HEAD 파일

		파일 수정 후 git add 후 스테이징에 올린걸 되돌린다.
		확인 : git status

	* 깃 버전 2.23 부터 작업 되돌리기 명령어가 변경 되었다.
		가장 최신 버전으로 되돌리기
			ex ) git checkout -- test.txt ==> git restore test.txt
		스테이징 되돌리기
			ex ) git reset HEAD test.txt ==> git restore --staged test.txt
	
	3.	최신 커밋 되돌리기
		git reset HEAD^

	4.	특정 커밋으로 되돌리기
		git reset --hard 커밋해시
		커밋 후 git log 를 입력 하면 커밋 해시를 구할 수 있다.

		/**	reset 명령어 옵션	**/
			--soft HEAD^  : 최근 커밋을 하기 전 상태로 작업 트리를 되돌립니다.
			--mixed HEAD^ : 최근 커밋과 스테이징을 하기 전 상태로 작업트리를 되돌립니다. 옵션없이 git reset 명령을 사용 할 경우 이 옵션이 기본.
			--hard HEAD^  : 최근 커밋과 스테이징, 파일 수정을 하기 전 상태로 작업트리를 되돌립니다. 이옵션으로 되돌린 내용은 복구할 수 없다.
	
	5.	커밋 삭제하지 않고 되돌리기
		git revert 커밋해시

	7. 깃과 브랜치
		- 웹 개발 중 고객이 새로운 기능을 요청했을 때 기존 파일에 새로운 기능을 위한 소스를 추가해서 새로운 버전을 만들기에는 버그나 문제가 있을 수 도 있다.
		- 제대로 동작하는 소스는 그대로 둔 채 새 소스를 추가한 버전을 따로 만들어 관리 하고 완벽하게 동작이 되면 기능을 추가한다.
		- 만약 한개의 소스가 여러 곳에 서비스 된다고 했을때 고객들 마다 원하는 기능이 달라 추가될 경우 소스를 여러개로 만들어야 하겠지만 그건 비효율적이고 관리 하기 힘들다.

		1. 브랜치 기능 살펴 보기
			-	깃으로 버전 관리를 시작하면 master 라는 브랜치가 만들어진다. 사용자가 커밋할 때마다 master 브랜치는 최신 커밋을 가리킨다 브랜치는 커밋을 가리키는 포인터와 비슷하다고 본다.
			-	master 브랜치에서 뻗어 나오는 새 브랜치를 분기 라고 한다.
			-	새 분기에서 작업을 했으면 브런치에 있던 파일을 원래 master 브랜치에 합칠 수 있는데 그것을 병합(merge) 라고 한다.

		2. 브랜치 만들기

			1. git branch "브랜치명"
				ex) git branch apple			
				1-1. 브랜치 생성 확인 git branch 명령어 로 확인할 수 있다.
				ex)
					$ git branch
					apple
					* master
				master 앞에 * 표시는 현재 master 브랜치 에서 작업하고 있다는 뜻이다.
				브랜치가 추가된 후에는 git log 도 다르게 나온다.
				ex)
				$ git log
				commit "커밋해시" (HEAD -> master, apple)
				저장소에 master 과 apple 가 있다는 뜻이고 HEAD->master 이므로 현재 작업중인 브랜치는 master 이라는 뜻이다.
			
			* 브랜치를 여러개 만든 후 커밋을 하면 
			ex)
			$ git log --oneline
			038f270 (HEAD -> master) work 4
			2af13dc (ms, google, apple) work 3
			79e4e7a work 2
			f2e3612 work 1
			로 나온다 work 3 에서 브랜치 3개를 추가했기 때문에 work3이 최신 커밋이고 아직 브랜치 작업은 master 이기 때문에 master 만 work4 커밋했다.
		3. 브랜치 사이 이동하기
			
			1. git checkout "브랜치명"
				ex)
				$ git checkout apple
				Switched to branch 'apple'

				브랜치 이동 후 git log 를 확인 하면 처음 분기 생성된 work3 커밋 그대로다 cat work.txt 파일을 확인해보면 work4 전인 work3 내용이 있다.
			* git log --oneline 한줄에 한 커밋씩 간략하게 확인

		4. 브랜치 정보 확인하기
			
			1. git log --oneline 한줄에 한 커밋씩 간략하게 확인

			2. git log --branches  각 브랜치의 커밋을 함께 볼 수 있다.

			3. git log --graph 그래프 형태로 확인.

		5. 브랜치 사이의 차이점 알아보기
			
			1. git log "기준점브랜치".."비교브랜치"	왼쪽 브랜치 기준으로 오른쪽 브랜치를 비교해서 알려준다.

		6.	브랜치 병합하기
			- 만들어진 브랜치에서 작업을 하다가 어느 시점에서는 작업을 마무리 하고 기존 master 브랜치에 병합하는 경우가 생긴다.

			1. 서로 다른 파일 병합하기
				-	git branch test / test 라는 새로운 브랜치 생성
				-	git checkout test / test 브랜치로 이동
				-	vi test.txt / 파일 생성
				-	git add -> git commit -m "test파일 생성" / 커밋
				-	git checkout master / 병합을 위해 마스터 브랜치로 이동.
				-	git merge test  / 마스터 브랜치에 test 브랜치를 가져와서 병합하기.

			2. 같은 문서의 다른위치를 수정했을 때 병합하기
				-	vi test.txt / 파일 생성
				-	git add -> git commit -m "test파일 생성" / 커밋
				-	vi test.txt / master 브랜치에서 파일 선 수정
				-	git branch test / test 라는 새로운 브랜치 생성
				-	git checkout test / test 라는 브랜치로 이동
				-	vi test.txt / test 브랜치에서 파일 후 수정
				-	git checkout master / master 브랜치로 이동
				-	git merge test / 병합

			3. 같은 문서의 같은 위치를 수정했을 때 병합하기
				-	vi test.txt / 파일 생성
				-	git add -> git commit -m "test파일 생성" / 커밋
				-	vi test.txt / master 브랜치에서 파일 선 수정
				-	git branch test / test 라는 새로운 브랜치 생성
				-	git checkout test / test 라는 브랜치로 이동
				-	vi test.txt / test 브랜치에서 같은 위치 내용 수정 
				-	git checkout master / master 브랜치로 이동
				-	git merge test / 병합		
				-   이전에 병합 했을 때 처럼 편집기가 열리지않고 에러 문구가 나온다.
					자동 병합하는 동안 충돌이 났다는 뜻이다.		
					충돌이 생긴 문서는 직접 편집 해줘야된다.
					vi test.txt 후 파일 내용을 보면
					<<<<<<<<<<<HEAD , >>>>>>>>>>>>>test 부분이 있는데 문서를 수정한후 지워줘야된다.
					저장 후 다시 git commit -am "충돌부분 해결" 하고 커밋하면 된다.
			
			4. 병합이 끝난 브랜치 삭제하기
				-	git branch  / 브랜치 확인
				-	git checkout master / 기본 브랜치는 master 이므로 삭제는 master 에서 해야된다.
				-	git branch -d test / test 브랜치를 삭제하기 위해서는 -d 옵션을 사용한다.
					
					*	브랜치 삭제후 다시 test 브랜치를 생성하면  예전에 작업했던 내용들이 그대로 나타난다.
						브랜치를 삭제하는것은 잠시 숨겨두는거라고 생각하면 된다.

		7. 브랜치 관리하기
2022-01-14
			-	책을 한번 다 살펴보고 다시 확인하자.
			1. git checkout  
				-	생성한 브랜치로 이동하는 명령어
					ex) git branch test  후 git checkout test
			2. git reset
				-	특정 커밋으로 되돌린다.
					ex) git log 후 git reset 커밋해시
			3.	git stash
				-	수정중인 파일 감추기 및 되돌리기
					파일을 수정하고 있는 도중 급하게 다른 파일을 수정해서 커밋해야 할 경우 사용한다. 
					ex) git status 를 입력하면 modified  에 수정중인 파일 이름이 나온다.
					그 다음 git stash  를 입력하면 수정전 상태로 돌아간다.
					감춰둔 파일을 확인 하는 명령어는 git stash list 이고 스택형식으로 쌓인다.
					급한 작업을 마무리 한 후 감춰둔 파일을 꺼내오는 명령어는 git stash pop  이다 git stash 하기 전으로 돌아온다.

					순서 : 
							수정 중 다른 파일 수정
							git stash -> 다른파일 수정 -> git stash list -> git stash pop -> 기존 수정 파일 수정 완료 후 커밋

				- git stash apply
					-	git stash pop 처럼 임시 저장한 내용을 가져오지만 리스트에서 삭제 시키지 않는다.
				- git stash drop
					-	stash 목록에서 가장 최근 항목을 삭제합니다.
